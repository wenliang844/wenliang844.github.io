视频智能侦测系统：从采集到正式告警的链路重构

June 16, 2026

围绕“采集 -> 算法 -> 规则 -> 正式告警”重整后端结构，让智能分析链路从能跑变成好维护、好扩展、好迁移。

JavaSpring BootMaven 多模块ONNX RuntimeRabbitMQ

今年最值得复盘的一件事，是把视频智能侦测系统里分散的采集服务、算法服务、规则引擎和业务告警入口重新梳理成一个更清晰的后端工程。它不是单纯换目录，而是重新确认数据应该在哪里产生、在哪里沉淀、在哪里被解释。

重构重点

  • 模块边界：用 Maven 父工程拆出公共能力、采集、算法、规则和业务域模块，减少跨模块直接依赖。
  • 数据分层：将待分析任务、标准分析结果、兼容事实和正式告警分层落库，避免历史表结构继续吞掉新业务语义。
  • 算法平台化：支持算法注册、规则绑定算法、执行器插件化和统一结果写入，为人脸、姿态、目标检测、OCR 等能力留扩展口。
  • 迁移策略：新链路基于标准分析结果运行，旧事实链路保留兜底，降低重构期间的业务风险。

这次经验给我的最大提醒是：复杂系统的重构不只追求“新架构漂亮”，更重要的是把链路上的关键名词固定下来。命名稳定了，接口、表结构和排查路径才会慢慢稳定。

One of the most valuable things to review this year was reorganizing the scattered capture service, algorithm service, rule engine and business alert entry point in the video intelligence system into a clearer backend project. This was not just a directory reshuffle; it was a chance to decide where data should be produced, persisted and interpreted.

Refactor Focus

  • Module boundaries: Split shared capabilities, capture, algorithms, rules and business domains under a Maven parent project to reduce direct cross-module dependencies.
  • Data layering: Persist analysis tasks, standard analysis results, compatibility facts and formal alerts in separate layers, so legacy table structures no longer absorb new business meaning.
  • Algorithm platformization: Support algorithm registration, rule-to-algorithm binding, pluggable executors and unified result writing, leaving extension points for face, pose, object detection, OCR and other capabilities.
  • Migration strategy: Run the new pipeline on standard analysis results while keeping the old fact pipeline as a fallback to reduce business risk during the refactor.

The biggest reminder from this work is that refactoring a complex system is not just about making the new architecture look elegant. It matters more to stabilize the key terms across the pipeline. Once naming becomes stable, APIs, table structures and troubleshooting paths gradually become stable too.

规则引擎与告警闭环：让分析结果变成可解释事件

May 28, 2026

把算法输出组织成规则运行时上下文，支持窗口聚合、事实时间线和统一告警创建。

规则引擎告警系统RedisMySQL

算法结果本身只是“看见了什么”，规则引擎要回答的是“这件事是否值得通知业务”。今年在规则链路上做的核心工作，就是把标准分析结果转换为可判断、可追踪、可解释的运行时上下文。

设计思路

  • 事实时间线：按点位、算法、目标和时间组织事实，让连续命中、间隔命中和缺失判断都能被表达。
  • 窗口聚合：在规则侧处理短时间内的多次识别结果，减少重复告警，也保留必要的证据链。
  • 入口收口：统一自研算法、海康事件和第三方接入的告警创建流程，让查询、统计、导出共享同一套业务口径。
  • 兼容兜底：旧事实链路在迁移期继续可用，保证新旧规则能逐步切换。

我更喜欢把规则引擎看成“业务解释层”，它不能只会跑表达式，还要能说明为什么命中、命中了什么、后续由哪个业务域接住。

Algorithm results only describe "what was seen"; the rule engine has to answer whether the event is worth notifying the business about. The core work on the rule pipeline this year was turning standard analysis results into runtime context that can be judged, traced and explained.

Design Ideas

  • Fact timeline: Organize facts by point, algorithm, target and time, so continuous hits, interval hits and missing conditions can all be represented.
  • Window aggregation: Process multiple recognition results in a short time window on the rule side, reducing duplicate alerts while preserving the evidence chain that still matters.
  • Unified entry point: Consolidate alert creation for in-house algorithms, Hikvision events and third-party integrations, so queries, statistics and exports share the same business definition.
  • Compatibility fallback: Keep the old fact pipeline available during migration, allowing old and new rules to switch gradually.

I prefer to think of the rule engine as a business interpretation layer. It should not only run expressions; it should also explain why a rule matched, what matched, and which business domain should handle the follow-up.

财税 SaaS 后端实践：报表、搜索与通知中心

April 18, 2026

围绕财务报表计算、ElasticSearch Starter 和多渠道通知中心，沉淀可复用的业务基础能力。

Spring Boot StarterElasticSearchRocketMQEasyExcelCompletableFuture

财税 SaaS 的复杂度不在“接口很多”，而在金额口径、租户隔离、异步处理和外部渠道都要求稳定。后端模块按 API、Implement、Runtime、Starter 分层后，很多通用能力可以从业务代码里抽出来。

实践记录

  • 财务报表：基于凭证行、会计期间、科目借贷方向和辅助核算计算科目余额表、利润表、现金流量表，金额统一使用 BigDecimal。
  • 搜索 Starter：封装索引初始化、Mapping、批量写入、别名切换、scroll 分页、聚合查询和高亮回填，降低业务接入成本。
  • 通知中心：支持短信、邮件、站内信、App Push 和微信公众号模板消息，多租户上下文与发送结果统一落库。
  • 并发发送：公众号批量发送前预加载模板和 OpenId 绑定信息，再用 CompletableFuture 和线程池隔离耗时操作。

这类系统很适合训练“业务口径优先”的后端思维：先把数据定义对，再谈性能优化和组件复用。

The complexity of a finance SaaS system is not that it has many APIs. It is that amount definitions, tenant isolation, asynchronous processing and external channels all have to remain stable. After layering backend modules into API, Implement, Runtime and Starter, many shared capabilities can be extracted from business code.

Practice Notes

  • Financial reports: Compute account balance sheets, income statements and cash-flow statements from voucher lines, accounting periods, debit/credit direction and auxiliary accounting dimensions, with all amounts represented by BigDecimal.
  • Search Starter: Encapsulate index initialization, mapping, batch writes, alias switching, scroll pagination, aggregation queries and highlighted result mapping to lower integration cost.
  • Notification center: Support SMS, email, in-app messages, App Push and WeChat Official Account template messages, with tenant context and send results persisted in a unified way.
  • Concurrent sending: Preload template and OpenId binding information before bulk Official Account sends, then isolate time-consuming operations with CompletableFuture and thread pools.

Systems like this are good training for backend thinking that starts from business definitions: define the data correctly first, then talk about performance optimization and component reuse.

低代码平台：Schema、物料与出码链路

March 22, 2026

从设计器、物料协议到 Web Worker 出码，把页面搭建能力组织成可扩展的平台工程。

TypeScriptReactMobXLowCode SchemaWeb Worker

低代码平台最有意思的地方，是它同时包含编辑器交互、协议设计、工程生成和运行时渲染。做这类项目时，前后端边界会变得更开放，也更考验抽象是否克制。

链路拆解

  • 设计器核心：围绕 designer、editor-core、editor-skeleton 实现画布搭建、节点选中、拖拽编排、面板扩展和插件注册。
  • Schema 渲染：将页面描述转换成 React/Rax 可运行视图，覆盖属性、插槽、条件渲染、循环渲染和数据源绑定。
  • 物料体系：扫描组件源码并提取属性、类型定义和元信息，生成中后台搭建所需的 JSON Schema。
  • 出码能力：支持浏览器端 Web Worker 出码和服务端出码，满足预览、下载和工程化交付。

低代码不是把代码藏起来，而是把重复的工程决策模板化。真正有价值的部分，是让业务页面更快落地，同时保留工程可维护性。

The most interesting part of a low-code platform is that it includes editor interaction, protocol design, project generation and runtime rendering at the same time. On this kind of project, frontend/backend boundaries become more open, and restrained abstraction matters even more.

Pipeline Breakdown

  • Designer core: Build canvas composition, node selection, drag-and-drop orchestration, panel extension and plugin registration around designer, editor-core and editor-skeleton.
  • Schema rendering: Convert page descriptions into runnable React/Rax views, covering props, slots, conditional rendering, loop rendering and data-source binding.
  • Material system: Scan component source code, extract props, type definitions and metadata, and generate the JSON Schema needed for middle/back-office page building.
  • Code generation: Support browser-side Web Worker code generation and server-side code generation for preview, download and engineering delivery.

Low-code is not about hiding code. It is about turning repeated engineering decisions into templates. The valuable part is helping business pages land faster while preserving maintainability.

Activiti 审批流：轻量 BPM 引擎封装

February 24, 2026

基于 Activiti 7 封装流程部署、任务运行、变量管理和流程图能力，让业务审批更容易接入。

Java 11Spring BootActiviti 7BPMN 2.0MyBatis

审批流项目的目标不是做一个“大而全”的 BPM 平台，而是把业务系统最常用的流程能力整理成稳定、轻量、可接入的工程模块。

封装范围

  • 多模块结构：拆分 API、核心引擎、公共组件和示例工程，避免业务直接感知引擎内部细节。
  • Starter 接入：通过 Spring Boot Starter 支持 BPMN 资源自动加载、流程部署和运行时配置。
  • 任务能力：封装任务查询、办理、变量管理、流程实例状态和业务状态同步。
  • 质量保障：引入 Checkstyle、JaCoCo、Surefire/Failsafe，让工作流这种状态复杂的模块更可控。

流程引擎最容易被低估的是“状态一致性”。监听器、业务表状态和流程实例状态必须有明确边界，否则排查会非常痛苦。

The goal of the approval-flow project was not to build a large, all-in-one BPM platform. It was to organize the workflow capabilities most commonly needed by business systems into a stable, lightweight and easy-to-integrate engineering module.

Wrapper Scope

  • Multi-module structure: Split API, core engine, shared components and sample projects, so business code does not need to know internal engine details directly.
  • Starter integration: Use a Spring Boot Starter to support automatic BPMN resource loading, process deployment and runtime configuration.
  • Task capabilities: Wrap task query, handling, variable management, process-instance state and business-state synchronization.
  • Quality guardrails: Introduce Checkstyle, JaCoCo and Surefire/Failsafe so a state-heavy module like workflow remains more controllable.

The easiest thing to underestimate in a process engine is state consistency. Listeners, business-table state and process-instance state must have clear boundaries, otherwise troubleshooting becomes painful very quickly.

工程化习惯：多模块、Starter 与质量工具链

January 12, 2026

把项目中反复出现的能力沉淀为公共组件、自动配置和检查流程，让交付质量更稳定。

MavenSpring Boot StarterDockerJenkinsJaCoCo

今年的几个项目都反复证明一件事：工程化不是额外负担，而是复杂业务能持续迭代的底座。多模块、Starter、自动化测试和代码检查，最终都是为了减少隐性沟通成本。

沉淀方向

  • 公共组件：把搜索、通知、导入导出、流程接入等重复能力抽成可复用模块。
  • 自动配置：通过 @ConfigurationProperties 和 AutoConfiguration 暴露统一接入方式，业务侧只关心参数和接口。
  • 数据库适配：在 Kingbase/PostgreSQL 等环境里处理 SQL 函数、分页、字段类型和变更脚本兼容。
  • 质量工具：用 Checkstyle、JaCoCo 和 CI 流程把代码风格、测试覆盖和构建结果固定下来。

我现在越来越倾向于把“可维护性”拆成具体动作：边界清楚、命名稳定、配置统一、失败可排查。它们比宏大的架构词更可靠。

Several projects this year repeatedly proved one thing: engineering practice is not extra overhead; it is the foundation that lets complex business keep evolving. Multi-module design, Starters, automated tests and code checks ultimately reduce hidden communication cost.

Directions for Reuse

  • Shared components: Extract repeated capabilities such as search, notification, import/export and workflow integration into reusable modules.
  • Auto-configuration: Expose a unified integration style through @ConfigurationProperties and AutoConfiguration, so business code only needs to care about parameters and interfaces.
  • Database adaptation: Handle SQL functions, pagination, field types and migration script compatibility in Kingbase/PostgreSQL and similar environments.
  • Quality tooling: Use Checkstyle, JaCoCo and CI flows to make code style, test coverage and build results explicit.

I increasingly prefer to break "maintainability" into concrete actions: clear boundaries, stable naming, unified configuration and diagnosable failures. They are more reliable than grand architecture words.
