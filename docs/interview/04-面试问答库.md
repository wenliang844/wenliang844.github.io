# 面试问答库

## 1. 架构设计

### Q1：这个项目整体架构是什么？

答：  
项目是父工程多模块架构，核心模块包括公共模块、采集服务、算法服务、规则引擎服务和业务域服务。主链路是“采集 -> 算法识别 -> 标准事实落库 -> 规则引擎判定 -> 正式告警”。采集服务负责从摄像头、海康或统一视频平台获取素材；算法服务把素材转换成标准事实；规则引擎按场所、点位角色和规则条件判断是否命中；业务域负责配置管理、告警创建、核验和展示。

### Q2：为什么拆成采集、算法、规则、业务域？

答：  
因为变化点不同。采集层面对设备和平台差异；算法层面对模型和推理性能；规则层面对业务条件和时间窗口；业务域面对配置、告警和处置。如果混在一起，新增算法或规则会影响全链路。拆分后每层职责清楚，问题也更容易定位。

### Q3：这个项目算微服务吗？

答：  
它具备微服务拆分实践，但不是完整 Spring Cloud 体系。采集、算法、规则、业务域和 Python 算法服务可以独立启动，服务之间通过 HTTP、RabbitMQ、数据库任务表和配置化地址协同。当前没有使用 Dubbo/Nacos，固定内网部署场景下这种轻量方式更可控。

### Q4：核心数据模型是什么？

答：  
核心是标准事实模型：`t_analysis_record` 和 `t_analysis_target`。前者表示一次算法分析记录，后者表示目标明细，比如人、手机、人脸、身份、行为、事件、置信度和目标框。规则引擎基于标准事实构造上下文，避免直接依赖某个算法的私有输出。

### Q5：为什么还保留旧的 `t_abnormal_record`？

答：  
项目处于重构过渡期。标准事实表是新主链，但历史展示、回退逻辑和旧链路仍依赖 `t_abnormal_record`。所以当前是“新链路优先、旧链路兼容”，不适合一次性删除旧表。

## 2. Spring Boot

### Q6：Spring Boot 在项目里承担什么？

答：  
负责服务启动、配置加载、MVC 接口、Bean 管理、事务管理和中间件集成。算法服务、规则服务、采集服务、业务域服务都有独立启动类和配置文件。

### Q7：项目里有哪些典型 Controller？

答：  
`CollectionTaskTriggerController` 提供抓图、清理、海康同步；`AlgorithmTaskTriggerController` 提供初始化分析和历史清理；`RulePendingTaskTriggerController` 提供规则任务处理、指标和死信重入；`WarningOpenApiController` 提供统一告警开放接口。

### Q8：`@PostConstruct` 有什么使用场景？

答：  
采集服务 `CapturePictureService` 在启动后初始化线程池并触发抓图任务。Python FastAPI 服务也在 startup 时调用算法预热，降低首次推理延迟。

## 3. MyBatis / JPA / 数据库

### Q9：为什么同时使用 JPA、MyBatis、MyBatis Plus？

答：  
这是历史系统重构常见情况。算法服务中有 JPA Repository，采集和业务域中有 MyBatis Plus Mapper 和 MyBatis XML。当前重构重点是先统一主链路和数据语义，再逐步治理持久层技术栈。

### Q10：多实例下如何避免分析记录重复？

答：  
任务层用 `claimToken` 领取任务，并支持超时回收和重试。结果层通过 `t_analysis_record.idempotency_key` 唯一约束保证标准事实幂等落库。

### Q11：规则任务表为什么需要索引？

答：  
规则任务主要按状态和重试时间领取，按 `bucket_key` 做分桶保序，按 `analysis_id` 查询分析批次。所以 `rule_pending_task` 有 `task_key` 唯一约束，以及 claim、bucket、analysis 相关索引。

### Q12：Liquibase 的作用是什么？

答：  
管理重构期数据库变更，例如创建 `rule_pending_task`、增加 `t_analysis_record.idempotency_key`、调整场所和规则字段。它让数据库变更可追踪、可重复执行，适合多环境部署。

## 4. Redis 与缓存

### Q13：Redis 用在哪里？

答：  
规则引擎用 Redis 保存规则运行态，核心类是 `RuleRuntimeStateStore`。状态 key 前缀默认 `znfx:rule-state`，TTL 是 7200 秒。同时使用 Redis `SET NX EX` 做分桶锁，避免多实例并发处理同一规则窗口。

### Q14：为什么规则运行态不用本地缓存？

答：  
规则服务可能多实例部署，同一场所和规则的事实可能被不同实例处理。运行态如果只在本地，会状态割裂。Redis 跨实例共享，更适合连续异常和窗口聚合。

### Q15：Caffeine 用在哪里？

答：  
算法服务用 Caffeine 缓存热配置和调度结果，`AlgorithmDispatchService` 还有短 TTL 结果缓存。Caffeine 适合短生命周期、读多写少、跨实例一致性要求不高的数据；Redis 适合规则运行态这种必须跨实例共享的数据。

### Q16：算法结果缓存怎么设计？

答：  
配置为 `algorithm.dispatch.result-cache.enabled=true`，TTL 15000 ms，最大 2048 条。它用于短时间内同一素材重复触发时复用结果，降低重复推理成本。TTL 短是为了降低配置或参数变化带来的一致性风险。

## 5. MQ / 可靠消息

### Q17：项目用的是 RocketMQ 吗？

答：  
不是，实际使用 RabbitMQ。公共模块有 `RabbitmqConfiguration`、`RabbitTemplateConfiguration` 和 `RabbitmqConstant`。RabbitMQ 用于算法事实事件触发和告警通知。

### Q18：RabbitMQ 在项目里做什么？

答：  
算法事实创建后发送到 `znfx.fact.analysis.exchange`，队列是 `znfx.fact.analysis.queue`，routing key 是 `fact.analysis.created`。规则服务消费后触发扫描。告警链路还会发送报警通知消息。

### Q19：为什么说 MQ 只是触发，DB 是兜底？

答：  
算法结果落库后会创建 `rule_pending_task`。MQ 消息只是通知规则服务尽快处理，真正要处理什么以数据库任务为准。即使 MQ 消费失败，任务还在 DB，可以通过定时扫描或手动触发继续处理。

### Q20：消费失败怎么办？

答：  
规则任务有状态、重试次数、下次重试时间和死信状态。失败后记录 `last_error`，未超过最大重试次数则重试，超过后进死信，后续可以通过接口重新入队。

## 6. Dubbo / Nacos / 服务治理

### Q21：项目用了 Dubbo 吗？

答：  
没有。项目服务间没有 Dubbo RPC，而是通过 HTTP、RabbitMQ 和数据库任务表协同。Java 调 Python 是 HTTP `/api/v1/analyze`，规则事实触发是 RabbitMQ，可靠处理以 DB 任务表为准。

### Q22：项目用了 Nacos 吗？

答：  
没有看到 Nacos 配置。当前服务地址主要通过配置管理，例如远程 Python 地址 `algorithm.remote.base-url=http://127.0.0.1:18080`。如果服务数量扩大或需要动态扩缩容，可以引入 Nacos 做注册发现和配置中心。

### Q23：没有注册中心有什么风险？

答：  
服务实例动态变化不够灵活，服务发现、灰度、熔断治理能力弱。当前项目适合固定部署，后续规模扩大后可以补 Spring Cloud、Nacos、OpenFeign、Sentinel 等。

## 7. Elasticsearch / 搜索

### Q24：项目用了 Elasticsearch 吗？

答：  
没有在主链路看到 Elasticsearch。当前告警和事实查询主要基于 MySQL。后续如果告警量大、需要复杂检索和聚合，可以把正式告警或事实摘要同步到 ES。

### Q25：如果引入 ES，你会怎么设计？

答：  
MySQL 作为事务主库，ES 作为查询索引。告警创建成功后通过 MQ 或 binlog 同步到 ES，索引字段包括告警 ID、场所、点位、规则、时间、部门、状态、身份和事件编码。查询走 ES，详情和处置回 MySQL。

## 8. 事务与一致性

### Q26：算法结果和规则任务如何保证一致？

答：  
标准事实落库在事务内完成。规则任务创建和事实事件发送放在事务提交后，使用 `TransactionSynchronizationManager.afterCommit()`，保证消费者看到的是已提交数据。即使 MQ 发送失败，DB 任务仍可被扫描处理。

### Q27：为什么 MQ 发送要 afterCommit？

答：  
如果事务未提交就发 MQ，规则服务可能先消费消息，但数据库还查不到事实，导致误判或空处理。`afterCommit` 可以保证消息触发时事实已经可见。

### Q28：正式告警怎么避免重复？

答：  
`StreamingWarningService` 创建告警时调用告警保存服务，返回结果里会标识是否重复。重复时跳过或复用已有告警；持续命中则追加证据和更新结束时间。

## 9. 性能优化

### Q29：项目有哪些性能优化？

答：  
采集侧用 `Semaphore(50)` 控制抓图并发，用独立 DB 线程池隔离写入。算法侧有 6 个分析线程、背压水位 64/24、YOLO 全局并发 3、同源图片并发去重和 15 秒结果缓存。规则侧批量领取 300、6 worker、单轮最多 3000 条或 60 秒。业务域 Druid 连接池 maxActive 为 50。

### Q30：算法服务为什么要做背压？

答：  
算法推理是 CPU/GPU 密集型，如果任务无限拉取，会造成线程池堆积、GPU 过载、数据库连接耗尽。背压高低水位控制任务进入速度，让系统高峰期保持可恢复。

### Q31：为什么限制 YOLO 全局并发？

答：  
YOLO 推理资源消耗大，同时跑太多会造成显存、CPU 或线程资源抖动。`algorithm.yolo.global.max-concurrent-runs=3` 是为了保护推理资源，让整体吞吐更稳定。

### Q32：规则服务为什么限制单轮处理数和耗时？

答：  
规则任务持续产生，如果一次扫描无限处理，会长时间占用线程，影响后续调度和故障恢复。`maxProcessCount=3000`、`maxRunMillis=60000` 让处理有明确边界。

## 10. 可靠性

### Q33：任务失败怎么处理？

答：  
算法任务和规则任务都有状态机。失败后记录错误、增加重试次数，未超过上限进入待重试，超过进入死信。处理中任务超时后可以恢复处理。

### Q34：算法服务挂了任务会丢吗？

答：  
不会直接丢。任务领取后有 `claimToken` 和处理超时时间。实例挂掉后任务仍在 DB，超时恢复逻辑可以重新处理。

### Q35：规则服务多实例同时处理同一场所规则怎么办？

答：  
通过 `bucketKey` 识别同一规则分桶，并用 Redis 锁保护运行态。拿不到锁的实例等待或跳过，避免同一窗口状态被并发修改。

## 11. 扩展性

### Q36：新增一个算法要改哪些地方？

答：  
配置算法注册表中的 `algorithmCode`、`entryKey`、版本和状态，在规则里绑定算法，然后新增一个 `AlgorithmExecutor` 实现类做输入适配、模型调用和标准输出映射。如果是 Python 模型，还要在 Python `AlgorithmRouter` 注册对应服务。

### Q37：新增业务规则怎么做？

答：  
优先通过业务域配置规则、规则条件和场所绑定。现有条件支持聚合、过滤和逻辑组合。如果规则语义全新，再扩展条件类型或规则评估服务。

### Q38：如何接入第三方告警？

答：  
调用统一告警开放接口 `/openapi/warnings/create`，携带 token、traceId、告警来源和渠道。服务端转换为 `CreateWarningCommand` 后走统一告警门面，避免第三方直接写正式告警表。

## 12. 代码质量与设计模式

### Q39：项目有哪些设计模式？

答：  
有策略/插件化模式：`AlgorithmExecutor` + `AlgorithmExecutorRegistry`；门面模式：`WarnVerificationCommandFacade`；应用服务编排：`PlaceRuleConfigApplicationService`；任务表模式：`rule_pending_task`；事件触发 + DB 兜底模式；配置化规则模式：`RuleBusinessConditionService`。

### Q40：项目有哪些风险点？

答：  
新旧链路并存，标准事实和历史事实还没完全收口；JPA、MyBatis、MyBatis Plus 混用；服务治理偏轻量；规则配置复杂后需要规则解释和回放工具；Python 模型依赖 GPU 和本地环境，部署稳定性需要加强。

### Q41：你会怎么继续优化？

答：  
第一，继续把规则事实主链完全切到标准事实表。第二，增加规则回放和命中解释。第三，补全 traceId 和采集-算法-规则-告警全链路指标。第四，服务规模扩大后引入注册中心、熔断限流和配置中心。

## 13. Trantor / 低代码平台

### Q42：项目用了 Trantor 吗？

答：  
没有看到 Trantor 相关依赖或配置。项目配置管理主要是自研业务域模块，如 `PlaceRuleConfigController`、`AlgorithmRegistryController`、`AnalysisRuleController`。如果面试官问低代码平台，我会说明本项目没采用 Trantor，但规则和算法配置已经做了一定平台化。

### Q43：如果用低代码平台改造，哪些部分适合？

答：  
适合后台配置页面，例如算法注册、规则条件、场所配置、摄像头角色绑定。不适合把核心规则引擎和算法编排完全低代码化，因为它们涉及任务幂等、时序状态、推理适配和可靠性处理。
