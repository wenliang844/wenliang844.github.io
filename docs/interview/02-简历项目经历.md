# 简历项目经历

## 版本一：Java 后端 / 平台服务方向

**项目名称：重点场所智能视频分析预警平台**

**项目描述：**  
该项目面向审讯室、接警大厅、服务窗口等重点场所，建设“采集 -> 算法识别 -> 标准事实落库 -> 规则引擎判定 -> 正式告警”的智能预警后端链路。系统由采集服务、算法服务、规则引擎服务、业务域服务和 Python 算法服务组成，支持多来源摄像头抓拍、YOLO/人脸/姿态/OCR 等算法识别、场所规则配置、告警生成和业务处置。

**技术栈：**  
Java 8、Spring Boot、Spring MVC、Spring Data JPA、MyBatis/MyBatis Plus、MySQL、RabbitMQ、Redis、Caffeine、Liquibase、Druid、Micrometer、FastAPI、OpenCV、ONNX Runtime GPU。

**个人职责：**

- 参与后端多模块重构，将采集、算法、规则、业务域能力拆分为职责清晰的服务模块。
- 设计并落地标准分析结果模型，将算法输出统一沉淀到 `t_analysis_record`、`t_analysis_target`，降低规则层对算法内部结构的耦合。
- 参与算法任务消费链路建设，通过 `claimToken`、超时回收、重试、死信和幂等键解决多实例并发处理下的重复分析问题。
- 参与规则引擎链路建设，基于 `rule_pending_task`、RabbitMQ 触发和 Redis 运行态实现规则任务可靠处理和窗口状态判断。
- 参与场所规则配置能力建设，支持“场所 + 点位角色 + 规则 + 算法绑定”的配置化预警能力。

**核心成果：**

- 将原有多仓库能力收敛为 5 个一级模块，形成可独立构建、独立部署、逐步替换的后端架构。
- 算法服务通过 `AlgorithmExecutor` 插件化机制支持 YOLO、人脸、姿态、OCR 等算法扩展，新增算法主要改适配层，不影响主链路。
- 规则服务单轮最多处理 3000 个任务，批量领取 300，6 个 worker 并发处理，并支持任务超时恢复和死信重放。
- 算法链路增加短 TTL 结果缓存，TTL 15 秒、最大 2048 条，降低同一素材短时间重复分析成本。
- Redis 规则运行态 TTL 7200 秒，并通过分桶锁保证同一场所/点位/规则窗口状态一致。

## 版本二：智能分析 / AI 工程化方向

**项目名称：智能视频算法分析与规则预警平台**

**项目描述：**  
项目通过 Java 算法编排服务和 Python 推理服务协同，实现对摄像头抓拍图片的人脸识别、目标检测、姿态行为和 OCR 识别，并将模型输出转换为标准事实，供规则引擎进行业务预警判断。系统重点解决算法结果不统一、模型调用耗时高、重复分析、规则与算法耦合等工程问题。

**技术栈：**  
Spring Boot、RestTemplate、Caffeine、MySQL、RabbitMQ、FastAPI、Pydantic、OpenCV、InsightFace、YOLO、ONNX Runtime GPU。

**个人职责：**

- 参与 Java/Python 算法服务解耦设计，Java 侧负责任务调度、幂等和事实落库，Python 侧负责模型推理和目标输出。
- 参与统一算法接口设计，Python 服务提供 `/api/v1/analyze`，按 `algorithmCode` 路由到人脸、YOLO、姿态、OCR 等具体算法。
- 参与算法执行器插件化，基于 `entryKey -> AlgorithmExecutor` 注册表实现算法适配层扩展。
- 参与算法结果标准化，将目标框、身份、行为、事件、置信度、原始 JSON 等信息统一映射到标准目标结构。
- 参与算法性能优化，包括任务背压、同源图片并发去重、YOLO 全局并发限制、模型预热和结果缓存。

**核心成果：**

- 建立 Java 编排 + Python 推理的跨语言算法架构，减少业务服务对模型框架和 GPU 依赖的直接耦合。
- 算法服务配置分析线程池 6、任务超时 600 秒、背压高低水位 64/24，提升高峰任务处理稳定性。
- YOLO 全局并发限制为 3，避免 GPU/CPU 推理资源被瞬时打满。
- Python 服务支持启动预热算法模型，降低首次识别延迟和冷启动不确定性。
- 标准事实模型为后续多算法融合规则提供统一数据基础。

## 版本三：规则引擎 / 消息可靠性方向

**项目名称：智能预警规则引擎与可靠任务处理系统**

**项目描述：**  
项目规则引擎负责消费算法事实，按场所、点位角色、规则条件和时间窗口判断是否生成正式告警。规则链路采用 RabbitMQ 事件触发与数据库任务表兜底相结合的方式，支持任务幂等、批量领取、重试、死信、超时恢复和 Redis 运行态管理。

**技术栈：**  
Spring Boot、RabbitMQ、Redis、MySQL、MyBatis/JPA、Liquibase、Caffeine。

**个人职责：**

- 参与 `rule_pending_task` 规则任务模型设计，支持 `task_key` 幂等、`claim_token` 领取、状态机、重试次数和死信状态。
- 参与 RabbitMQ 事实事件消费，采用“MQ 触发扫描，DB 保证可靠”的模式降低消息丢失风险。
- 参与规则运行态设计，使用 Redis 保存连续异常和窗口规则状态，TTL 7200 秒。
- 参与分桶锁设计，对同一 `bucketKey` 加锁，避免多实例并发处理同一规则窗口造成状态错乱。
- 参与规则条件引擎实现，支持 COUNT、EXISTS、NOT_EXISTS、AND/OR、身份/性别/行为/事件过滤等配置化条件。

**核心成果：**

- 规则服务支持每批领取 300 条任务、6 个 worker 并发、单轮最多 3000 条或 60 秒处理上限。
- 通过 `idx_rule_pending_claim`、`idx_rule_pending_bucket`、`uk_rule_pending_task_key` 等索引和唯一约束，提高领取、保序和幂等能力。
- 规则引擎从硬编码判断逐步转向“场所 + 点位角色 + 标准事实 + 配置条件”的可扩展模型。
- 告警创建链路支持重复拦截、证据追加和 MQ 通知，提升正式告警可靠性。

## 版本四：政企/业务域 SaaS 后端方向

**项目名称：重点场所智能预警业务域平台**

**项目描述：**  
项目业务域负责场所、摄像头、点位角色、规则、算法、告警和第三方接入等配置与业务能力，为智能视频预警平台提供统一管理后台和开放告警入口。系统内部包含设备管理、分析策略、告警核验、海康平台、第三方接入等多个子模块。

**技术栈：**  
Spring Boot、Spring MVC、MyBatis/MyBatis Plus、Spring Data JPA、MySQL、Druid、Liquibase、RabbitMQ、Redis、Micrometer/Prometheus。

**个人职责：**

- 参与业务域模块拆分，将设备管理、规则策略、告警核验、第三方接入等能力拆分到独立子模块。
- 参与场所规则配置服务，实现摄像头角色绑定、规则绑定、规则条件校验、ROI 和预览地址同步。
- 参与算法注册和规则绑定管理，支持前端配置算法编码、入口、版本、状态和规则算法关系。
- 参与统一告警开放接口建设，通过 `X-OPENAPI-TOKEN`、traceId、告警来源和渠道参数接入第三方告警。
- 参与 Liquibase 数据库变更管理，保障重构期表结构演进可追踪、可重复执行。

**核心成果：**

- 业务域工程包含约 75 个 Controller、107 个 Service，支撑配置、设备、告警、第三方接入等复杂业务。
- 形成统一的场所规则配置主线，降低新增场所、新增规则时的后端改动成本。
- 开放告警入口记录请求量、延迟和幂等命中指标，为后续监控和运营分析提供基础。
- Druid 主/从连接池最大活跃连接 50，适配后台管理和告警查询等高频数据库访问场景。

## 简历书写建议

优先选择其中一个版本作为主项目，不建议把四个版本全部写进同一份简历：

- 投 Java 后端：用版本一。
- 投 AI 工程化/智能平台：用版本二。
- 投中间件/高并发/可靠性方向：用版本三。
- 投政企 SaaS/业务平台后端：用版本四。

建议保留 4 到 6 个量化点：5 个一级模块、规则批量领取 300、6 worker、单轮最多 3000、Redis TTL 7200 秒、算法缓存 15 秒/2048 条、YOLO 并发 3、业务域 75 个 Controller 和 107 个 Service。
