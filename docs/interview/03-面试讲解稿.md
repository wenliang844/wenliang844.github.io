# 面试讲解稿

## 3 分钟版本

我这个项目是一个重点场所智能视频分析预警平台，主要面向审讯室、接警大厅、服务窗口这类需要实时风险识别的场景。平台核心链路是：采集服务从摄像头或第三方平台获取图片，算法服务做人员、人脸、手机、姿态等识别，规则引擎根据场所、点位角色和业务规则判断是否命中，最后生成正式告警给业务系统展示和处置。

整体架构上，项目拆成了 5 个一级模块：公共模块、采集服务、算法服务、规则引擎服务和业务域服务。业务域里面又拆了设备管理、分析策略、告警核验、海康接入等子模块。它不是一个单纯的 CRUD 系统，而是一个多服务协同、跨 Java 和 Python 的后端平台。

我重点参与的是主链路重构和稳定性建设。算法侧，我们把旧的待分析记录转换成标准分析任务，用 `claimToken` 做任务领取，用超时回收、重试和死信保证任务不会卡死。算法执行不是在调度器里写死 if/else，而是通过 `AlgorithmExecutor` 注册表按 `entryKey` 找到具体执行器；Java 侧负责任务编排、幂等和落库，Python FastAPI 服务负责模型推理，比如人脸、YOLO、姿态和 OCR。

规则侧，我们没有直接依赖 MQ 消费后马上判断，而是做了 `rule_pending_task` 任务表。RabbitMQ 只负责触发扫描，真正处理以数据库任务表为准，这样消息失败也能重试和补偿。规则引擎会基于 `t_analysis_record`、`t_analysis_target` 构造标准事实上下文，再结合场所、摄像头角色和规则条件判断是否生成告警。

性能和可靠性方面，算法服务有线程池、背压高低水位、同源图片并发去重和 15 秒短 TTL 结果缓存；规则服务每批可以领取 300 个任务，6 个 worker 并发，单轮最多处理 3000 个任务；规则运行态存在 Redis，TTL 是 7200 秒，并用分桶锁保证同一场所规则的状态一致。

这个项目的业务价值是把原来分散的采集、算法、规则和告警能力收敛成统一链路。后续新增算法、新增场所规则或者第三方告警接入时，不需要到处改历史代码，而是通过标准事实、规则配置和统一告警入口来扩展。

## 5 分钟版本

我做的项目可以概括为重点场所智能视频分析预警平台，业务上解决的是关键区域的异常行为识别和告警问题。比如审讯室里是否单人审讯、是否无人看管，服务窗口是否离岗，或者抓拍图片中是否有人、是否有手机、是否识别到民警身份。这些都不是普通后台表单逻辑，而是要把摄像头数据、AI 算法、业务规则和正式告警串起来。

系统整体是父工程多模块架构，一级模块包括 `znfx-shared-common`、`znfx-collection-service`、`znfx-algorithm-service`、`znfx-rule-engine-service` 和 `znfx-business-domain-service`。业务域服务里又有设备管理、分析策略、告警核验、第三方接入、海康平台等子模块。采集、算法、规则、业务域可以独立启动，算法服务默认端口是 8380，规则服务默认端口是 8480，Python 算法服务端口是 18080。

主链路是这样的：采集服务从统一视频平台、海康平台或者 NVR 获取图片，把素材写成待分析任务；算法服务把待分析记录构建成标准 `AnalysisTask`，然后多线程领取任务进行分析；算法结果统一落到 `t_analysis_record` 和 `t_analysis_target`；事务提交后创建规则待处理任务并发送 RabbitMQ 事实事件；规则服务扫描 `rule_pending_task`，构造场所规则上下文，判断命中后写正式告警月表。

我觉得项目里比较关键的一个设计是标准事实模型。旧链路里很多逻辑会直接依赖历史事实表或者算法输出字段，扩展新算法时容易牵一发动全身。现在算法输出会统一转换成标准分析记录和目标明细，里面包含算法编码、目标类型、身份、事件、置信度、目标框、原始 JSON 等信息。规则引擎不再直接关心某个算法内部怎么输出，而是基于标准事实做判断。

第二个重点是算法执行器插件化。算法服务里有 `AlgorithmExecutor` 接口和 `AlgorithmExecutorRegistry` 注册表，调度器通过 `entryKey` 找执行器。像 YOLO、人脸、姿态、OCR 都可以作为执行器接入。远程 Python 算法通过 `RemoteAlgorithmExecutor` 调 `/api/v1/analyze`，Python 侧再用 `AlgorithmRouter` 根据 `algorithmCode` 分发到具体模型。这样 Java 保持业务稳定性，Python 专注模型推理，两边通过统一接口契约交互。

第三个重点是可靠任务处理。算法任务和规则任务都不是简单查出来就处理，而是有领取令牌、状态流转、重试、超时恢复和死信。规则引擎的 `rule_pending_task` 表有 `task_key` 幂等键、`bucket_key` 分桶键、`claim_token`、`retry_count`、`max_retry_count` 和 `last_error`。RabbitMQ 只做触发，DB 任务表做可靠兜底，所以即使 MQ 消费失败，也可以靠任务表继续扫描恢复。

性能方面也做了不少控制。算法服务分析线程池是 6，任务超时 600 秒，背压高水位 64、低水位 24，YOLO 全局并发限制是 3，避免推理资源被打满。同一张源图片用 `activeSourceImageIds` 做并发去重，短时间重复分析结果用本地缓存，TTL 15 秒，最大 2048 条。规则服务每批领取 300 个任务，6 个 worker 并发，单轮最多 3000 个任务或 60 秒。

规则引擎这块的业务抽象是“场所 + 点位角色 + 规则条件 + 标准事实 + 时间窗口”。比如审讯室可能有嫌疑人位、民警位和值守位，不同点位在同一规则里承担不同角色。`RuleConditionEvaluationService` 支持 COUNT、EXISTS、NOT_EXISTS，也支持 AND/OR、身份、性别、行为、事件编码过滤。对于连续异常或者窗口规则，运行态存在 Redis，TTL 7200 秒，并且按 `bucketKey` 加锁，保证多实例下同一规则窗口状态不会被并发改乱。

最后业务域提供统一告警入口 `/openapi/warnings/create`，通过 token、traceId、来源和渠道来接入外部告警。内部规则命中也会通过 `StreamingWarningService` 创建正式告警，写月表、保存证据附件、发送通知 MQ。这个项目最终的成果是把多个旧系统的能力收敛成一条可扩展、可重放、可观测的智能预警后端链路。

## 10 分钟版本

我先从业务背景讲。这个项目是一个重点场所智能视频分析预警平台，主要服务于审讯室、接警大厅、服务窗口、枪库这类风险敏感场景。业务目标不是单纯展示摄像头画面，而是把抓拍图片或视频帧经过算法识别，转换成可被业务规则理解的事实，然后判断是否需要生成正式告警。例如“嫌疑人在场但民警离开超过一定时间”“窗口岗位无人”“画面里出现手机”“识别到非授权人员”等。

项目来源上，它不是一个全新小项目，而是对多个旧后端项目的重构收口。旧系统里采集、算法、规则和告警逻辑比较分散，表语义也有历史包袱。重构后的目标是明确一条主链路：采集服务负责让素材进入系统，算法服务负责把素材转成标准事实，规则引擎负责根据配置判断告警，业务域负责配置管理、正式告警和处置展示。

架构上，父工程有 5 个一级模块。`znfx-shared-common` 是公共基础层，里面有 RabbitMQ 配置、公共 DTO、数据库属性和 HTTP SDK。`znfx-collection-service` 是采集服务，负责抓图、同步海康配置、清理历史数据。`znfx-algorithm-service` 是算法编排服务，负责分析任务构建、任务领取、算法调度和标准事实落库。`znfx-rule-engine-service` 是规则引擎，负责处理规则任务、维护运行态并创建告警。`znfx-business-domain-service` 是业务域聚合工程，里面有 admin-web、设备管理、分析策略、告警核验、海康平台、第三方接入等子模块。

采集链路里，核心类是 `CapturePictureService` 和 `CollectionTaskTriggerController`。采集来源不是单一摄像头，而是统一视频平台、海康 AI 平台和直连 NVR。服务会按 `NORMAL`、`MAJOR`、`SPECIAL` 这几类任务拆分线程池，并通过 `Semaphore(50)` 控制抓图并发，避免同时对外部视频平台或网络 IO 造成过大压力。抓图结果会写入待分析记录，语义是“素材已经准备好，但还没有变成事实”。

算法链路里，第一步是 `AnalysisTaskBuildService` 把旧的待分析记录转换为标准 `AnalysisTask`。这里会用确定性 ID，避免同一条来源记录重复构建任务。第二步是 `AnalysisTaskProcessService` 领取任务，写入 `claimToken`，处理完成后更新状态；如果任务超时，可以回收重试；失败次数过多则进入死信。这个设计是为了支持算法服务多实例部署时不会重复抢同一批任务。

真正调算法时，核心是 `AlgorithmDispatchService`。它不会把所有算法调用硬编码在一个大方法里，而是根据规则绑定关系找到算法注册信息，再通过 `entryKey` 从 `AlgorithmExecutorRegistry` 里找到具体执行器。当前执行器包括 YOLO、人脸、姿态、OCR，以及远程 Python 执行器。这个设计的好处是新增算法时主要补一个执行器和输出映射，不需要改业务主链路。

Java 和 Python 的边界也比较清楚。Java 算法服务负责任务、事务、幂等、缓存、规则衔接；Python FastAPI 服务负责模型加载和推理。Python 服务统一提供 `/api/v1/analyze`，请求里有 `analysisId`、`algorithmCode`、`cameraId`、`source`、`roi`、`params` 等字段。Python 的 `AlgorithmRouter` 根据算法编码分发到人脸、YOLO、姿态、OCR 服务，并支持启动预热，降低首次推理延迟。

算法结果不是直接变成告警，而是先落标准事实表。`StandardAnalysisPersistenceService` 会写 `t_analysis_record` 和 `t_analysis_target`，并计算幂等键。数据库迁移里也能看到 `t_analysis_record.idempotency_key` 的唯一约束，目的就是避免多算法实例或重试导致重复事实。事务提交以后，再创建规则待处理任务，并发送 RabbitMQ 事实事件。

规则引擎是这个项目的另一个重点。它没有把 RabbitMQ 当作唯一可靠来源，而是做了 `rule_pending_task` 表。RabbitMQ 消息由 `FactAnalysisEventReceiver` 消费，但消息的作用主要是触发扫描，真正要处理什么以 DB 任务为准。`rule_pending_task` 有 `task_key` 唯一键、`bucket_key` 分桶键、`status`、`retry_count`、`claim_token`、`last_error` 等字段。这样规则任务可以批量领取、失败重试、超时恢复、死信重放。

规则判断不是简单 if/else。核心抽象是“场所 + 点位角色 + 规则 + 标准事实 + 时间窗口”。例如一个审讯室可能有嫌疑人点位和民警点位，单人审讯或无人看管不是看单个摄像头，而是看场所内不同角色点位在同一个时间窗口里的事实状态。`PlaceRuntimeRuleContextService` 会从标准事实表构造运行时上下文，`RuleConditionEvaluationService` 支持 COUNT、EXISTS、NOT_EXISTS、AND/OR、身份、性别、行为、事件编码等条件。

对于连续异常，规则引擎需要跨多轮事实维护状态。项目用 Redis 的 `RuleRuntimeStateStore` 保存规则运行态，默认 TTL 是 7200 秒。同时按 `bucketKey` 加 Redis 锁，锁时间 30 秒，获取超时 20 秒，避免多实例同时处理同一个场所/规则分桶导致状态乱序。这个点非常适合面试展开，因为它体现了规则引擎不是无状态接口，而是流式、窗口化、有并发一致性要求的服务。

缓存设计也分层。算法服务有 Caffeine 本地缓存，用来缓存热配置和调度结果；算法结果缓存 TTL 15 秒、最大 2048 条，减少同一素材短时间重复推理；规则运行态用 Redis，因为它要跨实例共享；Python 服务通过模型预热降低冷启动延迟。这里我会强调缓存不是“能放 Redis 就放 Redis”，而是根据数据是否跨实例、是否强一致、生命周期多长来选择。

正式告警这块，规则命中后由 `StreamingWarningService` 创建告警。它会解析场所、主点位、规则、开始/结束快照和证据快照，写入 `t_abnormal_record_YYYY_MM` 月表，同时保存 `t_abnormal_meter` 证据附件，并发送报警通知 MQ。如果持续命中，会追加证据和更新结束时间；如果重复命中，会按 `WarningRecordService.SaveWarningResult` 做重复拦截或复用。

从性能和可靠性结果看，项目里有很多可量化配置：采集并发 `Semaphore(50)`，算法线程池 6，任务超时 600 秒，背压高低水位 64/24，YOLO 全局并发 3，算法结果缓存 TTL 15 秒，规则批量领取 300，规则 worker 6，单轮最多 3000 条，Redis 规则运行态 TTL 7200 秒，业务域 Druid 连接池最大活跃 50。这些不是凭空说的指标，而是代码和配置里能看到的工程约束。

最后讲一下项目成果。业务上，它把原来分散的采集、算法、规则和告警能力统一成一条主链路，支撑重点场所风险预警。技术上，它建立了标准事实模型、算法执行器插件化、可靠任务表、MQ 触发/DB 兜底、Redis 运行态、缓存背压、统一告警入口和 Liquibase 迁移机制。难点是新旧表和历史链路还并存，所以重构不能一刀切，只能在保证线上可运行的前提下逐步收口。这也是我在项目里收获比较大的地方：复杂业务系统的重构，核心不是追求技术名词，而是把数据语义、失败恢复、幂等边界和扩展点设计清楚。

## 追问转场话术

### 如果问“你负责哪块”

> 我主要围绕主链路做后端建设，重点是算法任务处理、标准事实落库、规则任务可靠消费和场所规则配置。具体到代码上，包括分析任务的领取和重试、算法执行器注册调度、标准事实和目标落库、规则待处理任务、Redis 规则运行态，以及业务域的场所规则配置和告警入口。

### 如果问“这个项目最大难点是什么”

> 最大难点是它不是一个单接口系统，而是一条跨服务、跨语言、跨新旧数据结构的链路。算法输出要标准化，规则判断要依赖时间窗口和点位角色，多实例处理还要保证幂等和可恢复。所以我会把难点拆成三个：标准事实模型、可靠任务处理、规则运行态一致性。

### 如果问“你怎么证明有性能优化”

> 我不会只说用了缓存，而是从几个配置和代码点讲：采集服务用 `Semaphore(50)` 控制抓图并发；算法服务有 6 个分析线程、64/24 背压水位、YOLO 全局并发 3、同源图片并发去重和 15 秒结果缓存；规则服务批量领取 300、6 worker、单轮最多 3000；规则运行态 Redis TTL 7200 秒。这些都是围绕避免任务堆积、重复推理和状态并发错乱做的。
