# 项目亮点包装

## 亮点 1：统一分析预警主链路

**业务价值：**  
把原来分散的采集、算法、规则和告警能力收敛为一条清晰链路，降低联调、排障和后续扩展成本。

**技术实现：**  
父工程拆成公共、采集、算法、规则、业务域 5 个一级模块。主链路是 `t_abnormal_record_temp -> t_analysis_record/t_analysis_target -> rule_pending_task -> t_abnormal_record_YYYY_MM`。

**可追问点：** 为什么不是单体？如何兼容旧链路？

**回答思路：** 强调这是渐进式重构，先统一数据语义和主链路，再逐步收口历史逻辑。

## 亮点 2：标准事实模型

**业务价值：**  
算法输出结构各不相同，标准事实模型让规则层不直接依赖某个算法的私有结果。

**技术实现：**  
算法结果统一落到 `t_analysis_record` 和 `t_analysis_target`，记录分析批次、算法编码、目标类型、身份、行为、事件、置信度、bbox 和原始 JSON。

**可追问点：** 标准事实和旧事实表是什么关系？算法失败怎么处理？

**回答思路：** 当前是新旧并存，标准事实是新主链，旧表承担兼容和兜底。

## 亮点 3：算法执行器插件化

**业务价值：**  
新增算法不需要改主调度链路，降低扩展成本。

**技术实现：**  
`AlgorithmExecutor` 定义执行器接口，`AlgorithmExecutorRegistry` 收集执行器，调度器通过 `entryKey` 找具体执行器。现有执行器包括 YOLO、人脸、姿态、OCR、远程 Python 执行器。

**可追问点：** 新增算法要改哪些代码？为什么不是完全零代码？

**回答思路：** 新算法通常仍需要输入适配、模型调用和输出映射，所以平台化边界是“注册 + 绑定 + 执行器插件化 + 标准落库”。

## 亮点 4：Java 编排与 Python 推理解耦

**业务价值：**  
Java 负责业务稳定性，Python 负责算法生态和 GPU 推理，模型升级不会直接冲击主业务链路。

**技术实现：**  
Java `RemoteAlgorithmExecutor` 调 Python `/api/v1/analyze`。Python `AlgorithmRouter` 根据 `algorithmCode` 分发到人脸、YOLO、姿态和 OCR 服务，并支持启动预热。

**可追问点：** Python 服务不可用怎么办？为什么不用 Java 跑所有模型？

**回答思路：** Java 保留本地和远程执行器两种模式；Python 更适合模型生态，Java 更适合事务、幂等和规则衔接。

## 亮点 5：分析任务多实例领取与幂等

**业务价值：**  
防止多实例重复分析、重复落库或任务卡死。

**技术实现：**  
`AnalysisTaskBuildService` 使用确定性任务 ID；`AnalysisTaskProcessService` 用 `claimToken` 领取任务，支持超时回收、重试和死信；`t_analysis_record.idempotency_key` 做唯一约束。

**可追问点：** 实例挂了怎么办？重试怎么避免重复？

**回答思路：** 领取解决并发，超时恢复解决宕机，唯一键解决重复落库。

## 亮点 6：MQ 触发 + DB 任务表兜底

**业务价值：**  
规则判断不能因为 MQ 消息失败就漏告警，DB 任务表让处理可重试、可排查、可补偿。

**技术实现：**  
算法事实落库后创建 `rule_pending_task` 并发送 RabbitMQ。`FactAnalysisEventReceiver` 消费消息后触发扫描，但真正处理以 DB 任务为准。任务表含 `task_key`、`bucket_key`、`status`、`retry_count`、`claim_token`、`last_error`。

**可追问点：** 为什么不用纯 MQ？DB 任务表会不会慢？

**回答思路：** MQ 适合解耦和触发，DB 适合可靠状态管理；性能通过索引、批量领取和处理上限控制。

## 亮点 7：场所 + 点位角色 + 规则建模

**业务价值：**  
重点场所规则通常不是看单个摄像头，而是看一个场所内不同角色点位的组合状态。

**技术实现：**  
`PlaceRuleConfigApplicationService` 管理场所、摄像头关系、点位角色关系和规则关系。规则引擎通过 `PlaceRuntimeRuleContextService` 构造运行上下文。

**可追问点：** 点位角色有什么用？配置错误怎么避免？

**回答思路：** 角色让规则表达更贴近业务语义；保存时校验场所、摄像头角色、规则绑定、条件角色、重复绑定和 ROI。

## 亮点 8：规则条件配置化

**业务价值：**  
常见规则可以通过配置表达，减少硬编码和发版频率。

**技术实现：**  
`RuleBusinessConditionService` 管理规则条件，`RuleConditionEvaluationService` 支持 `COUNT`、`EXISTS`、`NOT_EXISTS`、`AND/OR`、身份、性别、行为、事件编码等过滤和聚合。

**可追问点：** 所有规则都能配置化吗？如何解释命中原因？

**回答思路：** 常见聚合和过滤规则可配置；复杂新语义仍需要扩展条件类型。后续可做规则回放和命中解释。

## 亮点 9：Redis 规则运行态与分桶锁

**业务价值：**  
连续异常和窗口规则需要跨多次事实保存状态，多实例下还要保证同一窗口状态一致。

**技术实现：**  
`RuleRuntimeStateStore` 用 Redis 保存状态，TTL 7200 秒；使用 `SET NX EX` 对 `bucketKey` 加锁，默认锁时间 30 秒。

**可追问点：** 为什么不用 MySQL？锁过期怎么办？

**回答思路：** Redis 适合高频短期状态，MySQL 适合持久事实和最终结果；锁过期避免死锁，TTL 避免状态无限增长。

## 亮点 10：性能优化与背压控制

**业务价值：**  
视频分析和模型推理资源消耗大，高峰期如果无限拉任务会拖垮服务。

**技术实现：**

| 位置 | 优化 |
| --- | --- |
| 采集 | `Semaphore(50)` 控制抓图并发 |
| 算法 | 线程池 `6`，背压水位 `64/24` |
| YOLO | 全局并发 `3` |
| 规则 | 批量领取 `300`，worker `6`，单轮最多 `3000` |
| 缓存 | 结果缓存 TTL `15s`，最大 `2048` |

**可追问点：** 为什么不直接加线程？水位怎么定？

**回答思路：** 算法瓶颈在 GPU/CPU/IO，不是线程越多越好；水位应结合压测和任务积压调优。

## 亮点 11：短 TTL 结果缓存

**业务价值：**  
同一素材可能被多规则或重试短时间重复触发，缓存可以降低重复推理成本。

**技术实现：**  
`algorithm.dispatch.result-cache.ttl-ms=15000`，`max-size=2048`。同时 `activeSourceImageIds` 防止同源图片并发重复处理。

**可追问点：** 缓存会不会影响准确性？

**回答思路：** TTL 很短，只覆盖重复触发窗口；缓存 key 应包含源图片、算法、ROI、参数等影响结果的因素。

## 亮点 12：统一告警网关

**业务价值：**  
第三方告警和自研规则告警逐步收口到统一正式告警模型，避免各处直写表。

**技术实现：**  
`WarningOpenApiController` 提供 `/openapi/warnings/create`，通过 `X-OPENAPI-TOKEN` 鉴权，支持 `X-TRACE-ID`、`X-ALARM-SOURCE`、`X-CHANNEL`。`WarningGatewayApplication` 转换为 `CreateWarningCommand` 后调用告警门面。

**可追问点：** 如何鉴权和追踪？

**回答思路：** token 鉴权、traceId 透传、来源/渠道记录、请求量和延迟指标，后续可增加签名、防重放和限流。

## 亮点 13：Liquibase 支撑渐进式数据库演进

**业务价值：**  
重构项目表结构变化多，Liquibase 让变更可追踪、可重复执行。

**技术实现：**  
`20260521-01-rule-pending-task.xml` 创建规则任务表和索引；`20260604-01-analysis-record-idempotency.xml` 增加标准分析记录幂等键。

**可追问点：** 和手写 SQL 有什么区别？

**回答思路：** changeSet 管理执行历史，配合 preConditions 避免重复执行，多环境更一致。

## 亮点 14：可观测性基础

**业务价值：**  
链路长，问题可能发生在采集、算法、规则、告警任一环节，指标和日志能降低排障成本。

**技术实现：**  
业务域开放 Actuator/Prometheus endpoint；`WarningGatewayApplication` 记录请求量、耗时和幂等命中；规则有 `/pendingMetrics`；算法和规则有慢日志和阶段耗时日志。

**可追问点：** 线上怎么定位告警没生成？

**回答思路：** 按素材是否采集、分析任务是否成功、标准事实是否落库、规则任务是否生成、规则是否命中、告警是否落库逐层排查。

## 亮点 15：清楚识别技术边界

**业务价值：**  
没有盲目堆中间件，而是按实际问题选型，避免面试时说过头。

**技术实现：**  
项目没有使用 Dubbo/Nacos/RocketMQ/Elasticsearch/Trantor。服务协同用 HTTP、RabbitMQ 和 DB 任务表；搜索主要基于 MySQL；配置管理由业务域自研。

**可追问点：** 为什么不用 Nacos、RocketMQ、ES？

**回答思路：** 当前固定部署和链路规模下轻量方案更简单可控；后续服务规模扩大或检索复杂度提高时再引入对应组件。

## 推荐优先讲的 5 个亮点

1. 标准事实模型。
2. 算法执行器插件化。
3. MQ 触发 + DB 任务表兜底。
4. Redis 规则运行态 + 分桶锁。
5. 性能优化与背压控制。
